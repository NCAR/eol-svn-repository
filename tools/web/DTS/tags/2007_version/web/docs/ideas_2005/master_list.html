Master List

The Master List (ML) will become just a viewer without a special ML editor as the tool is currently set up. It will use information that is set through the DTS. Flags like the in progress and new will be displayed depending on the values of certain fields in the dataset_project table. There will be flag to be set in the same table to include/discard a dataset from the list. The dataset will be placed into each category that it is associated with through the dataset_category table. This will allow a single edit of the dataset and it will be changed and displayed with each category in the list. (i.e. Altair Radiosondes will show up under both the Upper Air Soundings and the Ship Altair categories.)

Suggested Changes

  • Straight Database Access

    The current version(s) of the ML has the editor overwrite a static web page for the main list after every save. The category list is a set of pages (there are also pages for each category that has a subcategory) that are only hand editable (to my knowledge). These need to be updated when the categories are changed for the project and could be done easily if they were read when the database was read.

    I think that having the ML read from the database would be better even though it will probably be slower than a static web page. The biggest gain that I see is it would make the ML a single point for data entry. We would not have to know or remember where the static files were located because it was all taken care of through the database.

    It would also handle the category - subcategory relationship internally without having to create seperate pages. I have written some javascript that performs a similar function that (from an observational point of view) should be easy to adapt to our needs here. The one potentially large hurdle in this could be how different browsers may handle the javascript and making them all handle the subcategory insertion in a reasonable fashion.

    Another suggestion for this would be a seperate database read when expanding/contracting a category. This probably would be slower than the javascript, but would be more likely to behave in a consistant manner between browsers.

  • Handling Multiple Doc Files

    With the new design of jedi, there is now the potential of having several readmes, EULA's, and other documentation files for a dataset. The ML needs to be able to display these to the user.

  • Making Composites Stand Out More

    This is a specific issue that has come recently that some (many??) of the users are not aware of the composites that we form. An attempt should be made to make these more visible with possiblities of different naming conventions (include multi-network in the dataset title) and/or having a different background color for these datasets.

  • General Layout Redesign

    At this point, we may want to consider redesigning the entire ML layout and format. This may be necessary anyway to handle some of the other functionality we want to add to the tool.
