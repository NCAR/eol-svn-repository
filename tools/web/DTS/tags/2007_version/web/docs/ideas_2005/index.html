NATTAP 2005

NATTAP 2005 is the current name for the database and tools that will be used for managing our data through its life-cycle. It currently contains the DPG project pages, the Data Tracking System (DTS), the Master List (ML), and the Inventory Processing Tool (IVEN). Each tool (except for the DPG project pages) has its own database. The purpose of NATTAP is to combine these tools into a single entity (this can include multiple views) to make our life easier by maintaining the data once and in a single place.

The 2005 is used to denote the year of the proposed upgrade to prevent it from being confused with the earlier (and presumably later) definitions for NATTAP in general.

This will probably be a multiple step process. It should also be noted that this is not a completely decoupled process. The database will drive the interfaces and the interfaces will drive the database as well. The steps are provided to give a starting point and a process that may be able to be followed to complete the upgrade in a timely and satisfactory manner.

  1. Defining a Unified Database

    1. Defining all of the tables that should be included in NATTAP.
    2. Upgrading the current tools to use this new database.

  2. New Interface(s) to the Database

    1. Upgrading/Redesigning the old tools to better handle the new database and make things cleaner and easier to use.

General Data Management Discussion

This area is for general discussion of topics that are general to NATTAP or do not fit with an individual topic.

It would be a good idea to convert all of our tools into a single language whether it is Perl CGI or Java Server Pages (JSP). Currently, the DTS is in JSP, the ML is in static HTML and CGI, and IVEN is in CGI. I would think it would be better to go to CGI for two reasons.

  1. Our current group is very familiar with Perl and would be able to make changes as necessary without solely relying on the creator of the tool to perform maintainence. Going to JSP would require more learning on everyones part. I do not know if JSP would be faster or not than CGI which could cause a change of mind in that direction.
  2. JSP requires Tomcat to be installed and working on the server. It is currently setup on hurricane and was on gale before it was taken down. I do not know if it is on any others at this time. The biggest issues in my experience with Tomcat is that
    1. Every change to the program requires a restart to the server. This is extremely annoying and really slows down debugging. (This could have just been something wacky with my server, but it was how the instructions told me to set it up.)
    2. Tomcat requires the application to be under a webapps directory under the Tomcat installation. This is a pain (and can be seen by having to type in the dpg login everytime the DTS is started since it has a different .htaccess file because of its different location).
